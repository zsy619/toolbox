1. 提示工程是什么？

提示工程是设计高质量输入（prompt）以引导大语言模型产出精确输出的迭代过程。无需机器学习背景，但需要理解模型、配置、措辞与结构。不充分的提示会导致模糊、幻觉或无效回答。

💡 核心观点：LLM 本质是预测引擎，提示工程就是设定正确的“下一个 token”路径。

LLM 工作原理速览

模型基于训练数据，根据前面 token 预测下一个 token。输出配置（温度、Top-K/P）控制随机性与多样性。提示工程需结合模型能力与任务目标，通过优化提示及采样参数获得最佳结果。

2. 输出配置：温度 · Top-K · Top-P

参数 作用 推荐起始值
Temperature 随机性控制。0=贪婪解码（确定性）；越高越多样/创意 确定性任务:0.0 ~ 0.2；创意任务:0.7~1.0
Top-K 仅从概率最高的 K 个 token 中采样。K=1 等价贪婪 20~40 平衡；高K增加多样性
Top-P (nucleus) 累积概率不超过 P 的 token 集合。P=0.9~0.95常用 0.95 (连贯且适度创意)

📌 组合建议：
▪ 事实/数学/分类 → 温度=0, Top-K=1
▪ 创意故事 → 温度=0.9, Top-P=0.99, Top-K=40
▪ 平衡模式 → 温度=0.2, Top-P=0.95, Top-K=30
⚠️ 注意：过高的自由度可能导致重复循环或无关内容，需反复测试。

3. 核心提示技术 (含官方示例)

🎯 零样本 / 少样本提示 (Zero/Few-shot)

零样本：无示例，直接描述任务。如“将评论分类为正面/负面/中性”。少样本：提供1~5个高质量示例，引导输出格式与逻辑。尤其适合分类、结构化抽取。

📋 示例 (电影评论分类):
Prompt: “Review: 'Her' is a disturbing masterpiece. Sentiment:” → Output: POSITIVE
通过少样本加入更多边缘案例可提升鲁棒性。
🎭 系统 / 角色 / 上下文提示

系统提示：设定全局任务（例如“只返回 JSON”）。
角色提示：赋予模型身份，如“你是一名资深旅行向导，用幽默风格推荐景点”。
上下文提示：提供即时背景，例如“我正在写一篇关于80年代街机游戏的博客”。

💬 示例 (角色+幽默): “你作为旅行向导，以幽默风格推荐纽约3个必去地点。” → 模型生成风趣的景点描述。
🧠 思维链 (Chain of Thought) 及自洽性

CoT：引导模型逐步推理。例如 “Let's think step by step.”。大幅提升数学、逻辑准确率。
自洽性 (Self-Consistency)：多次采样 CoT 并多数投票，进一步提高可靠性。代价是更多 token。

# 示例：年龄问题
Prompt: “我3岁时，伴侣年龄是我的3倍。现在我20岁，伴侣多大？让我们一步步思考。”
→ 输出推理步骤：3岁→伴侣9岁，年龄差6岁，现在20岁→伴侣26岁。
🌲 思维树 (Tree of Thoughts)
允许多条推理路径并行探索，在复杂规划、创意写作等任务上超越线性 CoT。模型维护多个思维分支，通过搜索或评估选择最优路径。
⚡ ReAct (Reason + Act)

结合推理与外部工具（搜索引擎、API）。模型交替生成“思考 → 行动 → 观察”循环，实现智能体行为。

# Python + LangChain 示例（伪代码）
agent = initialize_agent(tools=["serpapi"], llm=llm, agent=AgentType.ZERO_SHOT_REACT_DESCRIPTION)
agent.run("Metallica乐队成员一共有多少孩子？")
→ 自动搜索成员信息，累加结果返回10。
🔙 后退一步提示 (Step-back)
先让模型回答一个更广泛、原理性的问题，再将答案作为上下文提升最终准确性。例如针对游戏关卡设计，先问“FPS游戏中常见的挑战性场景有哪些？”，再基于此生成剧情，显著提升创意质量。

4. 代码提示：编写 · 解释 · 翻译 · 调试

🔧 生成 Bash 重命名脚本
# Prompt: 写一段 Bash 代码，要求用户输入文件夹名，然后将文件夹内所有文件加上前缀 "draft_"
模型自动生成带注释的脚本，经过测试可正常运行。但是注意: LLM 生成的代码务必人工审查。
🐍 代码翻译 (Bash → Python)
# 通过提示词将上述 Bash 转为 Python，使用 os 和 shutil 模块实现相同功能
模型生成 Python 版本，包含错误处理、路径拼接。还给出改进建议（保留扩展名、处理空格）。
🐛 调试与代码审查
当你提供错误代码（如调用未定义的 toUpperCase），模型不仅指出错误，还重写为 prefix.upper()，并额外优化了文件扩展名保留、异常捕获。提示词示例：“以下 Python 代码报错 NameError，请调试并改进。”
LLM 响应：修复函数名，增加 try/except，使用 os.path.splitext 保留扩展名，提高健壮性。

5. 专家级最佳实践

📌 优先使用指令，而非约束

正面指令（“生成一篇3段落的博客”）比约束（“不要列出游戏名称”）更有效。指令清晰、正面引导可减少模型困惑。

📐 结构化输出: JSON + Schema

强制 JSON 格式减少幻觉，便于集成。结合 JSON Schema 预定义字段类型（string, number, date），让模型精准提取。但 JSON 较长可能截断，可使用 json-repair 库修复。

🧪 少样本分类时打乱类别顺序

避免模型记住示例顺序而非特征。提供 3~6 个混合类别的示例，提升泛化能力。

📝 记录每一次实验 (模板)

| 名称 | 目标 | 模型 | 温度 | Top-K/P | 提示全文 | 输出结果 | 评分 |

🎛️ 动态变量 & 适应模型更新

在提示中使用变量（例如 {city}）便于复用。当模型版本升级（如 Gemini 1.5 Pro 新特性），需重新评估并微调提示词以利用新能力。

🖼️ 多模态提示前沿

Gemini 等模型支持图像+文本输入，可进行视觉问答、图表推理等。多模态提示拓展了应用边界，结合图像与上下文可获得更丰富输出。
